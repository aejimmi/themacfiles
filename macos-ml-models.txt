macOS ML Models
===============
What they do. Where they live. How to reverse them.

Observed on macOS 15 Sequoia. All models are files on disk, loaded at runtime by
CoreML, compiled for Neural Engine / GPU / CPU. SIP prevents modification but not
reading. Everything here is readable without disabling SIP.

3. CORE RECOGNITION (com.apple.mediaanalysisd uses these for Live Text)
-----------------------------------------------------------------------
Framework: /System/Library/PrivateFrameworks/CoreRecognition.framework/Versions/A/Resources/
Format: .mlmodelc.bundle containing espresso files
Model count: 4 base models, localized variants per language

  cc_excnnfs_model          Latin-script character recognition (CNN + feature sequence)
  cc_chcnnfs_model          Chinese character recognition
  cc_cncnnfs_model          CJK character recognition
  cc_flat_model             General flat text model

Naming convention for localized variants (loaded by mediaanalysisd):
  cr_lw_en-US               CoreRecognition Live Words, English
  cr_lw_th-TH               CoreRecognition Live Words, Thai
  cr_td_model_v3            CoreRecognition Text Detection v3

The two-stage pipeline: cr_td detects text regions in the image, then cr_lw
recognizes characters within each region. One cr_lw model per supported language.


4. INTELLIGENCE PLATFORM (com.apple.intelligenceplatformd)
----------------------------------------------------------
Framework: /System/Library/PrivateFrameworks/IntelligencePlatformCore.framework/Versions/A/Resources/
Format: .mlmodelc bundles with metadata.json
Signal configs: 165 JSON files in Views/config/

This is the behavioral prediction engine. It builds a knowledge graph of your
contacts, locations, activities, and relationships. The models classify your
social graph and predict what's relevant to you in context.

Models:

  EntityTagging_Family.mlmodelc
    Type: XGBoost tree ensemble (xgboost 1.7.5 via coremltools 6.3.0)
    Output: binary — is this contact "family" or not
    Inputs: 130+ features (full list extracted from metadata.json):

    Contact metadata:
      isVIP, sameLastName, hasBirthday, hasNickname, isFavorite,
      iCloudFamilyMember, nameContainsEmoji, nicknameContainsEmoji,
      allowListedNameMatch, allowListedNicknameMatch

    Address and email:
      sharesHomeAddress, sharesWorkAddress,
      hasSameEmailDomains, hasSamePrivateEmailDomain,
      hasSamePublicEmailDomain, hasSameWorkEmailDomain,
      hasSameEduEmailDomain

    Photo-derived:
      photosPersonAge, ageDifferenceCategory,
      photosAgeDifferenceCategory

    Behavioral — how often you interact, by day and time:
      personOverallPopularity, personLongTermPopularity,
      personPopularityGiven{Monday..Sunday},
      {day}PopularityGivenPerson,
      personPosteriorProbabilityGiven{day},
      personPopularityGiven{Morning,Afternoon,Evening,Night},
      personPopularityGiven{Home,Work},
      {home,work}PopularityGivenPerson,
      personPopularityGiven{Weekday,Weekend}

    Photos behavioral (same temporal grid but from photo co-occurrence):
      photosPersonOverallPopularity, photosPersonLongTermPopularity,
      photosPersonPopularityGiven{day}, {day}PopularityGivenPhotosPerson,
      photosPersonPosteriorProbabilityGiven{day},
      photosPersonPosteriorProbabilityGiven{day}{timeOfDay} (7x4 = 28 slots),
      photosPersonPopularityGiven{Home,Work,Weekday,Weekend},
      hasWeekendWeekdayPhotosFeatures

    Interaction:
      top1PersonInteractionMechanism, top2PersonInteractionMechanism,
      top3PersonInteractionMechanism

  EntityTagging_FamilyAndFriends.mlmodelc
    Type: XGBoost tree ensemble
    Inputs: 144 features (same as Family + extras)
    Output: binary — is this contact "family or friend"

  The entity tagging system as a whole outputs probability scores for:
    mother, father, parent, sister, brother, sibling, family, friend,
    familyAndFriends, partner, coworker, alumni, child, son, daughter,
    myself, unknown
  Stored in: entityTagging/entityTagging.db, table "entity_tagging"
  Updated: every 2 hours from personEntityTaggingFacts, daily full refresh

  EntityRelevanceModel.mlmodelc
    Type: XGBoost tree ensemble
    Inputs: 30 features — all context-based:
      behaviorPopularityGivenContext:{coarseGeoHash, coarseTimeOfDay, dayOfWeek,
        loi, specificGeoHash}
      behaviorPopularityGivenSequence
      contextPopularityGivenBehavior:{coarseGeoHash, coarseTimeOfDay, dayOfWeek,
        loi, specificGeoHash, wifi}
      posteriorProbabilityGivenContext:{coarseGeoHash+coarseTimeOfDay,
        coarseTimeOfDay+dayOfWeek, coarseTimeOfDay, dayOfWeek, loi,
        coarseTimeOfDay+specificGeoHash, specificGeoHash,
        coarseTimeOfDay+wifi, dayOfWeek+wifi, wifi}
      dailyDoseL1Error, dailyDoseL2Error
      lastExecutionAge, medianTimeIntervalBetweenExecution
      longTermTrendingPopularity, shortTermTrendingPopularity, trendingPopularity
      sequencePopularityGivenBehavior
    Output: binary — is this entity relevant right now
    This model decides: given your current location (geohash), time of day,
    day of week, WiFi network, and location-of-interest — who/what matters
    to you in this moment.

  DiningOutModel.mlmodelc
    Type: neural net
    Inputs: POI (point of interest), PassKit (wallet/payment), FIA (food in activity)
    Predicts dining-out behavior from location + payment + activity signals.

  EntitySimilarityLinearModel.mlmodelc
    Linear model for computing similarity between entities (people, places).

  ETShadowModel.mlmodelc
    Shadow/test model for entity tagging (A/B testing or validation).

  LOITypeToOneHotTransformer.mlmodelc
    Converts location-of-interest type codes to one-hot feature vectors.

  Entity importance signals (entityImportanceSignals.db):
    Stores per-contact: isFavorite, isEmergencyContact, isICloudFamily,
    relationshipLabel. Sourced from: CNContact, FAFamily, HKEmergencyContact,
    IPEntityTaggingPersonInference.

Framework: /System/Library/PrivateFrameworks/IntelligencePlatform.framework/Versions/A/Resources/

  EntityRerankerModel.mlmodelc     Re-ranks entity relevance results
  locationEncoder.mlmodelc         Encodes locations into embedding vectors
  MDNameToEmailPersonLinker.mlmodelc   Links names to email addresses to people

RELATIONSHIP CLASSIFICATION
----------------------------
Every person Apple knows about — from contacts, photos, or both — gets scored
for 17 relationship categories. Updated every 2 hours. Works on nameless faces.

  mother, father, parent, sister, brother, sibling, family, friend,
  familyAndFriends, partner, coworker, alumni, child, son, daughter,
  myself, unknown

Each is a probability score 0.0-1.0. A face with no contact card, no phone
number, no email can still score family=0.317 from photo co-occurrence alone.

  Person type:      Ordinary, Celebrity (inferred), Pet (inferred)
  Detection type:   human, pet

Data: ~/Library/IntelligencePlatform/Artifacts/entityTagging/entityTagging.db
Dump: ./scripts/dump-entity.sh <id>


PER-FACE BIOMETRIC PROFILING
------------------------------
Every detected face in every photo gets a full physical description.
73 columns per detection. This is what Apple stores about each face:

  Gender:        unknown, male, female
  Age:           unknown, baby, child, teen, adult, senior
  Ethnicity:     unknown, type 1-4 (labels not decoded)
  Skin tone:     numeric scale (observed 1-3)
  Hair color:    unknown, black, brown, blonde, red, gray, other
  Hair type:     numeric codes (not yet decoded)
  Facial hair:   none, stubble, beard, other
  Glasses:       none, prescription, other
  Headgear:      none, type codes (labels not decoded — type 5 most common)
  Face mask:     no, yes
  Eye makeup:    none, type codes
  Lip makeup:    none, type codes
  Smile:         no/yes, slight/broad
  Eyes:          open/closed per eye
  Gaze:          direction type, angle, confidence, target coordinates
  Pose:          frontal, half-profile, angled, yaw, roll
  Expression:    type codes
  Quality:       0.0-1.0 face quality score + blur score
  Body:          upper body bounding box (center, width, height)
  Celebrity:     VIP model type code (celebrity detection)
  Is-me:         confidence score that this face is the device owner

Each detection links to: source photo filename, timestamp, GPS coordinates.
Each face cluster (ZPERSON) aggregates: display name, face count, merge
confidence, contact matching scores.

Data: ~/Pictures/Photos Library.photoslibrary/database/Photos.sqlite
Tables: ZDETECTEDFACE (73 cols), ZPERSON (30 cols)
Dump: ./scripts/dump-entity.sh <id>

Database column reference (for writing queries):

  ZGENDERTYPE, ZAGETYPE, ZETHNICITYTYPE, ZSKINTONETYPE, ZHAIRCOLORTYPE,
  ZHAIRTYPE, ZFACIALHAIRTYPE, ZGLASSESTYPE, ZHEADGEARTYPE, ZHASFACEMASK,
  ZEYEMAKEUPTYPE, ZLIPMAKEUPTYPE, ZHASSMILE, ZSMILETYPE, ZFACEEXPRESSIONTYPE,
  ZEYESSTATE, ZISLEFTEYECLOSED, ZISRIGHTEYECLOSED, ZGAZETYPE, ZGAZEANGLE,
  ZGAZECONFIDENCE, ZGAZECENTERX, ZGAZECENTERY, ZPOSETYPE, ZPOSEYAW, ZROLL,
  ZQUALITY, ZQUALITYMEASURE, ZBLURSCORE, ZBODYCENTERX, ZBODYCENTERY,
  ZBODYHEIGHT, ZBODYWIDTH, ZCENTERX, ZCENTERY, ZSIZE, ZVIPMODELTYPE,
  ZDETECTIONTYPE, ZPERSONFORFACE, ZASSETFORFACE, ZFACEPRINT, ZFACECROP


5. PEOPLE SUGGESTER
--------------------
Framework: /System/Library/PrivateFrameworks/PeopleSuggester.framework/Versions/A/Resources/
Model count: 10

  ContactRankerModel.mlmodelc
    Type: pipeline classifier
    Inputs: 35 features — ranks contacts by interaction likelihood:
      timeSinceLastContactVia{Share, IncomingText, OutgoingText, IncomingCall}
      normalized{Share, IncomingText, OutgoingText, IncomingCall, OutgoingCall}Frequency
      numberOfBehavioralRulesAdvocating
      averageBehavioralRule{Support, Confidence, Lift, Conviction, PowerFactor, MLScore}
      minimumBehavioralRule{Support, Confidence, Lift, Conviction}
    The "behavioral rules" are association-rule mining patterns: "when you're
    at work on Tuesday morning, you text person X." The model uses mined rule
    support/confidence/lift as features.

  interactionPrediction.mlmodelc
    Type: neural net classifier
    Predicts who you'll interact with next.

  iCloudFamilyModel_{gbdt,dt,rf,IP_gbdt}.mlmodelc
    Four model variants (gradient-boosted, decision tree, random forest, IP-boosted).
    Family detection from iCloud signals. Separate from the IntelligencePlatform
    EntityTagging models — these run on iCloud-derived features.

  compiledShareSheetModel.mlmodelc         Share sheet contact ranking
  sharesheetEmptyBaseModel.mlmodelc        Baseline when no history
  cloud_mlp_with_embedding.mlmodelc        Cloud-based MLP with embeddings
  psSuggestionsRankingModel.mlmodelc       Suggestion ranking (1 input: timeSinceLastOutgoingInteraction)

15. HOW TO INSPECT MODELS
--------------------------

Dump a .mlmodelc model spec (Python):

  pip install coremltools
  import coremltools as ct
  model = ct.models.MLModel("EntityTagging_Family.mlmodelc")
  spec = model.get_spec()
  print(spec.description)                    # inputs, outputs, metadata
  print(spec.treeEnsembleClassifier)         # full tree structure for XGBoost

Read .espresso.net directly (it's JSON):

  cat cnn_facepose.espresso.net | python3 -m json.tool
  Shows: layer types, connections, tensor shapes, activation functions.

Read model metadata:

  cat EntityTagging_Family.mlmodelc/metadata.json | python3 -m json.tool
  Shows: input schema (feature names and types), output schema, model type,
  version, coremltools version, source framework.

Visualize architecture:

  Open any .mlmodelc or .espresso.net in Netron (netron.app).
  Full computation graph with layer shapes and connections.

Extract XGBoost decision trees:

  spec = model.get_spec()
  trees = spec.treeEnsembleClassifier.treeEnsemble
  for tree in trees.nodes:
      print(tree.nodeBehavior, tree.branchFeatureIndex, tree.branchFeatureValue)
  This gives you every split: which feature, what threshold, which branch.
  Fully reconstructable as human-readable decision logic.

Class-dump the framework binaries:

  class-dump /System/Library/PrivateFrameworks/MediaAnalysis.framework/mediaanalysisd
  Reveals Objective-C class names: MAFaceRegionDetector, MASceneClassifier,
  MATextSafetyEvaluator, etc. Maps models to their runtime consumers.

Read the signal pipeline configs:

  ls /System/Library/PrivateFrameworks/IntelligencePlatformCore.framework/\
    Versions/A/Resources/Views/config/
  Each JSON file describes one signal: what data it reads, where it stores
  results, what triggers updates, and what other signals depend on it.
